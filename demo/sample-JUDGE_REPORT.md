## Top 3 Failure Modes

1. **"This has been built before" objection** *(Creativity: -5 pts)*
   A skeptical judge will immediately think of 3–5 existing standup generators. The differentiation — git history vs. form-filling — is real but undersold. The pitch needs to lead with the git insight, not the standup output.
   
   → **Fix**: Open the demo by showing the git log first. Say: "The information already exists — it's right here in your commits. Nobody is reading it." Then run the tool. Lead with the source, not the output.

2. **No agent personality** *(Agent Quality: -5 pts)*
   StandupBot's SOUL.md (if it existed) would be generic. "I am a helpful assistant that summarizes commits." There's no distinctive voice, no memorable persona, no reason for a judge to think about this agent after the demo.
   
   → **Fix**: Give the agent a voice. Make it opinionated about commit quality — have it call out vague commits ("fix stuff") and suggest better messages. That's a personality, not just a feature.

3. **Skills are thin for a hackathon** *(Skill Design: -8 pts)*
   One skill (summarize-commits) doesn't demonstrate composability. The skill design criterion rewards multiple focused skills that work together. With only one skill, you're leaving 8 points on the table.
   
   → **Fix**: Add a `coach-commits` skill that analyzes commit message quality, and a `weekly-summary` skill that generates a weekly report. Now you have a pipeline: analyze → summarize → report. Spend 45 minutes and recover most of those points.
