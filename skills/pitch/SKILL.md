# Pitch Skill

## Purpose
Turn what you've built into a story that wins. Generate a complete pitch package that is grounded in the actual codebase — not in what you wished you'd built.

This skill must be run AFTER the `evaluate` skill. A pitch without an adversarial check is a pitch that gets ambushed by judges.

## When to Use
- After `evaluate` skill has run and `JUDGE_REPORT.md` exists
- 1–2 hours before the submission deadline
- When the user asks "how do I present this?"
- When the README needs to be polished for judges to read

## Prerequisite Check
Before generating anything, verify:
1. `PROJECT_BRIEF.md` exists — if not, stop and ask user to run `ideate` first
2. `JUDGE_REPORT.md` exists — if not, stop and say: "Running pitch without an adversarial evaluation is dangerous. Run `evaluate` first."
3. Source code exists and is non-trivial — if the repo is mostly empty, flag it honestly

### Step 1: Activate Builder Mode
Switch to `[MODE: BUILDER]` and announce it.

### Step 2: Read All Context
Read in order:
1. `PROJECT_BRIEF.md` — the promise
2. `JUDGE_REPORT.md` — the weaknesses to pre-empt
3. `README.md` — current state of documentation
4. Source code entry point — what it actually does
5. `TODO.md` — what was completed vs. deferred

Note any gaps between what was promised and what was built. The pitch must bridge these gaps honestly.

### Step 3: Generate the Hook
Write a 1-sentence hook that:
- States the specific pain point in the user's language (not "leverages AI to optimize")
- Creates an emotional response ("have you ever...")
- Implies the solution without giving it away

Examples of good hooks:
- "Every hackathon team has great ideas at 2 AM — and a broken demo at 9 AM."
- "You've spent 18 hours building something. You have 2 minutes to make judges care."

### Step 4: Generate the 2-Minute Demo Script
Write `DEMO_SCRIPT.md` — a word-for-word spoken script for the live demo. Structure:

```
[0:00 – 0:20] THE HOOK
[Exact words to say — grabs attention in the first 10 seconds]

[0:20 – 0:45] THE PROBLEM
[State the problem with a real example. Make it specific. Make it relatable.]

[0:45 – 1:30] THE DEMO MOMENT
[Walk through the live demo step by step. Label each step clearly.]
Step 1: [What you click/type and what to say while doing it]
Step 2: [What happens next and why it matters]
Step 3: [The climax — the moment judges are impressed]

[1:30 – 1:50] THE INSIGHT
[Why this matters. What this enables. The vision beyond the demo.]

[1:50 – 2:00] THE CLOSE
[Strong, memorable final line. Not "any questions?" — something that sticks.]
```

### Step 5: Generate PITCH.md
Write `PITCH.md` to the current directory:

```markdown
# PITCH.md — [Project Name]

> Generated by HackFork Pitch Skill — [MODE: BUILDER]

## The Hook
[One sentence that opens the pitch]

## The Problem
[2-3 sentences. Specific. Real. Painful.]

## The Solution
[What the agent does, explained to a non-technical judge in 3 sentences]

## The Demo Moment
[Step-by-step description of the live demo — what to show, in what order, what to say]

## Why This Wins
[Against each judging criterion, one sentence on why this scores well]
- Agent Quality: ...
- Skill Design: ...
- Working Demo: ...
- Creativity: ...

## Likely Judge Questions & Coached Answers
**Q: "[Hardest question based on JUDGE_REPORT.md failure modes]"**
A: [Specific, confident answer that addresses the concern directly]

**Q: "[Second hardest question]"**
A: [Answer]

**Q: "[Third question — often about technical depth or scalability]"**
A: [Answer]

**Q: "[Question about differentiation from existing tools]"**
A: [Answer]

**Q: "[Question about what's next / business model]"**
A: [Answer]

## One-Line Closer
"[The last sentence judges will remember]"
```

### Step 6: Polish the README
Review the current `README.md` and suggest specific improvements so a judge reading it in 30 seconds understands:
1. What it does
2. How to run the demo
3. What makes it stand out

### Step 7: Pre-Empt the Weaknesses
For each failure mode in `JUDGE_REPORT.md`, write a "pre-emption statement" — a line the presenter can use to acknowledge the weakness before a judge raises it. Proactively addressing weaknesses eliminates their sting.

Example: "You might be wondering why we didn't add X — we consciously chose to focus on Y because that's what makes the demo compelling. X is a 2-hour addition post-hackathon."

## Output
- File: `PITCH.md` (written to current directory)
- File: `DEMO_SCRIPT.md` (written to current directory)
- README improvement suggestions
- Weakness pre-emption lines

## Quality Criteria
A good pitch from this skill:
- Has a hook that a non-technical judge would remember
- Has a demo script that works even if the live demo fails (includes graceful recovery)
- Has judge question answers that are confident, specific, and under 30 seconds each
- Has a closer that is NOT "any questions?" or "thank you for your time"
- Is grounded entirely in what was actually built — no exaggeration
