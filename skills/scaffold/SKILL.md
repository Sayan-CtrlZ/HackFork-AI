# Scaffold Skill

## Purpose
Turn a validated project brief into a real, runnable project skeleton. This skill generates the folder structure, key boilerplate files, and a prioritized TODO list — ordered by what matters most for the demo, not what's "cleanest" architecturally.

## When to Use
- Immediately after `ideate` skill completes
- When starting a new hackathon project from scratch
- When the current codebase structure is chaotic and needs reorganizing

## Instructions

### Step 1: Activate Builder Mode
Switch to `[MODE: BUILDER]` and announce it.

### Step 2: Read PROJECT_BRIEF.md
Read `PROJECT_BRIEF.md` from the current directory. If it doesn't exist, stop and ask the user to run the `ideate` skill first.

Extract:
- Project type (web app, CLI tool, API service, agent, etc.)
- Core demo moment
- Key features in scope
- Risk flags

### Step 3: Select Tech Stack
Based on the project type, recommend the fastest demo-ready stack. Default recommendations:

| Project Type | Recommended Stack |
|---|---|
| Web app | Next.js or Vite + React |
| AI agent | Node.js + Anthropic SDK or OpenAI SDK |
| CLI tool | Node.js + Commander.js |
| API service | Express.js or Fastify |
| Browser demo | Vanilla HTML/CSS/JS (zero build time) |
| Data/ML demo | Python + Streamlit or Gradio |

Ask the user to confirm the stack choice before generating files.

### Step 4: Generate Project Structure
Create the folder structure by writing all required directories and key files. For each file, write real content — not just comments.

**Always include these files:**
- `README.md` — Project name, one-liner, quickstart instructions
- `TODO.md` — Demo-impact ordered task list (see format below)
- `.gitignore` — Standard ignores for the chosen stack
- Entry point file (index.js, main.py, app.tsx, etc.) with real boilerplate

**Include based on project type:**
- `package.json` with correct dependencies (for Node projects)
- Environment template `.env.example` if any API keys are needed
- A `demo/` folder with sample inputs/outputs for the demo

### Step 5: Generate TODO.md
Write a `TODO.md` ordered strictly by **demo impact** (most impactful for the 5-minute demo first), NOT by technical dependency.

```markdown
# TODO.md — HackFork Build Plan

## Phase 1: Demo Core (must ship) 🔴
- [ ] [Task that makes the demo work, explained in plain English] — Est: Xh
- [ ] [Task that makes the demo impressive] — Est: Xh

## Phase 2: Demo Polish (should ship) 🟡  
- [ ] [Task that makes the demo smoother] — Est: Xh
- [ ] [Task that adds a wow moment] — Est: Xh

## Phase 3: Nice-to-Have (only if time allows) 🟢
- [ ] [Non-critical improvement] — Est: Xh

## Explicitly Deferred (do NOT build this hackathon)
- ~~[Feature that would blow the timeline]~~
- ~~[Feature that doesn't show in demo]~~

Generated by HackFork Scaffold Skill
```

### Step 6: Estimate Total Build Time
After generating TODO.md, sum the estimates and tell the user:
- Total estimated time: Xh
- If it exceeds their time budget: flag with `⚠️ Timeline risk — suggest which Phase 2/3 items to defer`

### Step 7: Suggest Next Skill
After scaffold completes: "Once you've built the Phase 1 items, run the **evaluate** skill to stress-test the project before pitching."

## Output
- `TODO.md` ordered by demo impact
- Project folder structure with real boilerplate
- Tech stack recommendation with rationale
- Build time estimate with risk flags

## Quality Criteria
A good scaffold from this skill:
- Has a `TODO.md` where every task is tied to the demo, not abstract tech debt
- Has a README that could be read by a judge in 30 seconds
- Has an `.env.example` if any secrets are needed (never commit real keys)
- Has a `demo/` folder with at least one sample input ready for the live demo
- Does NOT include any feature that was explicitly deferred in `PROJECT_BRIEF.md`
