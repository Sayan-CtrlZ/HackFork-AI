# Ideate Skill

## Purpose
Convert a fuzzy idea into a sharp, validated project concept — fast. This skill is the entry point of the HackFork pipeline. It must be run before any code is written.

## When to Use
- User has a rough idea but hasn't defined the problem clearly
- User is unsure which idea to pursue from multiple options
- User wants to understand how judges will perceive their concept
- Start of a hackathon session

## Instructions

### Step 1: Activate Builder Mode
Switch to `[MODE: BUILDER]` and announce it.

### Step 2: Run the 5 Socratic Questions
Ask the user these questions, one at a time (or all at once if they're in a rush). Wait for answers before continuing.

1. **"Who is the ONE specific person this is built for? Not 'users' or 'developers'. Name them. Describe their day."**
2. **"What is the worst thing that happens to that person if this doesn't get built?"**
3. **"What's the core feature that makes the 5-minute demo jaw-dropping? Not all features — just the ONE moment."**
4. **"What would a lazy team build in the same space? How is this different from that?"**
5. **"What can you ship in [hackathon duration]? Be brutally honest about scope."**

### Step 3: Apply the Constraint Filter
After getting answers, apply these filters:
- **Kill anything** that can't be demoed in 5 minutes
- **Kill anything** that requires data you don't have
- **Flag anything** that depends on a third-party API that could break during demo
- **Protect** the core demo moment at all costs

### Step 4: Run Judge Mode Pre-Check
Switch to `[MODE: JUDGE]` briefly. Score the idea against the 4 criteria on a scale of 1–5:
- Agent Quality potential: /5
- Skill Design potential: /5
- Working Demo potential: /5
- Creativity: /5

If total score is below 12/20, call it out explicitly and suggest improvements before proceeding.

### Step 5: Generate PROJECT_BRIEF.md
Write the following file to the current directory:

```markdown
# PROJECT_BRIEF.md

## Project Name
[Name]

## One-Line Pitch
[Pitch]

## Target User
[Specific user persona]

## Core Problem
[The specific pain point, stated in the user's language]

## Solution
[What the agent does, in plain English]

## Key Differentiator
[Why this beats the obvious alternative]

## The Demo Moment
[The ONE thing that will make judges put down their phones]

## Scope Boundaries
[What is IN scope for this hackathon]
[What is explicitly OUT of scope]

## Risk Flags
[Any demo risks: API dependencies, data requirements, time constraints]

## Judging Rubric Alignment
| Criterion | Score | Rationale |
|---|---|---|
| Agent Quality (30%) | /5 | ... |
| Skill Design (25%) | /5 | ... |
| Working Demo (25%) | /5 | ... |
| Creativity (20%) | /5 | ... |
| **Total** | **/20** | ... |

Generated by HackFork Ideate Skill
```

### Step 6: Suggest Next Skill
After generating `PROJECT_BRIEF.md`, tell the user: "Run the **scaffold** skill next to generate your project structure and prioritized TODO list."

## Output
- File: `PROJECT_BRIEF.md` (written to current directory)
- Mode switches: Builder → Judge → Builder
- Verbal summary: One paragraph confirming what was decided and what was protected

## Quality Criteria
A good PROJECT_BRIEF.md from this skill:
- Has a named, specific user (not "users")
- Has a single demo moment that can fit in 60 seconds
- Has a differentiation statement that isn't "it's AI-powered" (everything is)
- Has explicit out-of-scope items
- Has a rubric alignment table with honest scores
