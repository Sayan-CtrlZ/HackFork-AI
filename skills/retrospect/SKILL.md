# Retrospect Skill

## Purpose
Turn the hackathon into a learning machine. Analyze what worked, what failed, and why — then write a structured retrospective that makes the next hackathon better.

This skill is the only one that runs AFTER submission. It has no time pressure. Be thorough.

## When to Use
- After the hackathon ends (submitted or not)
- When the user wants to understand why they won or lost
- When preparing for the next hackathon
- When the user says "let's do a post-mortem"

### Step 1: Activate Builder Mode
Switch to `[MODE: BUILDER]` and announce it. For this skill, Builder Mode is reflective — not urgent. Slower, more analytical.

### Step 2: Collect All Evidence
Read in order:
1. `PROJECT_BRIEF.md` — What was the original plan?
2. `TODO.md` — What was planned in detail?
3. `JUDGE_REPORT.md` — What was the adversarial assessment?
4. `PITCH.md` — What was actually pitched?
5. `README.md` — What was actually documented?
6. Source code — What was actually built?
7. `git log --oneline` — The honest chronicle of what happened

### Step 3: Run the Gap Analysis
Compare:
- **Promised vs. Built**: Which features from `PROJECT_BRIEF.md` made it into the actual code?
- **Planned vs. Done**: Which tasks from `TODO.md` were completed?
- **Judge Report vs. Reality**: Did the actual demo address the `JUDGE_REPORT.md` failure modes?
- **Pitch vs. Code**: Was the pitch grounded in what was actually built?

### Step 4: Also Switch to Judge Mode Briefly
`[MODE: JUDGE]` — Review the overall execution:

Ask:
1. Did the team protect the core demo moment, or did they get distracted by peripheral features?
2. Did the team run evaluate before pitching, or did they skip it?
3. Were the failure modes from the judge report actually addressed?
4. What was the single decision that had the highest impact (positive or negative)?

### Step 5: Identify the 3 Skill Patterns
For each HackFork skill (ideate, scaffold, evaluate, pitch), assess:
- Was it used? At the right time?
- Did it produce the expected artifact?
- What would have happened if it was used earlier / later / differently?

### Step 6: Generate RETROSPECTIVE.md
Write the following file to the current directory:

```markdown
# RETROSPECTIVE.md — [Project Name]

> Generated by HackFork Retrospect Skill
> Hackathon: [Name if known]
> Date: [Date]

## What Was Built vs. What Was Planned
| Feature | Planned | Built | Notes |
|---|---|---|---|
| [Feature 1] | ✅ | ✅ | |
| [Feature 2] | ✅ | ❌ | Deferred — right call |
| [Feature 3] | ❌ | ✅ | Added mid-hackathon |

## TODO Completion Rate
- Phase 1 (Demo Core): X/Y tasks completed
- Phase 2 (Demo Polish): X/Y tasks completed
- Phase 3 (Nice-to-Have): X/Y tasks completed

## JUDGE_REPORT Remediation Rate
Of [N] failure modes identified:
- [N] were addressed before pitch ✅
- [N] were acknowledged but not fixed 🟡
- [N] were ignored ❌

## The Pivotal Moment
[The single decision — either the best decision made or the worst — that most affected the outcome. Be specific about what it was and what the alternative would have been.]

## What Worked Well
1. [Specific thing with evidence]
2. [Specific thing]
3. [Specific thing]

## What Failed and Why
1. **[What failed]**: [The root cause — not "we ran out of time" but WHY we ran out of time]
2. [Same]
3. [Same]

## HackFork Skill Analysis
| Skill | Used? | Timing | Artifact Generated | Effectiveness |
|---|---|---|---|---|
| ideate | ✅/❌ | [Time] | PROJECT_BRIEF.md ✅/❌ | [1-5] |
| scaffold | ✅/❌ | [Time] | TODO.md ✅/❌ | [1-5] |
| evaluate | ✅/❌ | [Time] | JUDGE_REPORT.md ✅/❌ | [1-5] |
| pitch | ✅/❌ | [Time] | PITCH.md ✅/❌ | [1-5] |
| retrospect | ✅ | post | RETROSPECTIVE.md ✅ | — |

## Top 3 Improvements for Next Hackathon
1. [Specific, actionable change to make next time — not generic advice]
2. [Same]
3. [Same]

## Quote from the Hackathon
"[The best or most revealing thing said during the hackathon — a moment that captures the experience]"

---
*This retrospective was generated by HackFork — the adversarial hackathon co-pilot.*
*Your git history is your truth. Everything else is narrative.*
```

### Step 7: Archive the Session (Optional)
If the user wants to preserve the hackathon history, suggest:
```bash
git checkout -b hackathon/[name]-[date]
git add PROJECT_BRIEF.md TODO.md JUDGE_REPORT.md PITCH.md RETROSPECTIVE.md
git commit -m "chore: archive hackathon session — [project name]"
```

This creates a permanent, searchable record of every hackathon the user runs HackFork on.

## Output
- File: `RETROSPECTIVE.md` (written to current directory)
- Optional: git archive branch creation command
- 3 specific improvements for next time

## Quality Criteria
A good retrospective from this skill:
- Has a gap analysis with real numbers (not "we mostly hit our targets")
- Identifies the *root cause* of failures (not surface symptoms)
- Has a "pivotal moment" insight that the team didn't see in the moment
- Has 3 improvements that are specific enough to be acted on at the next hackathon
- Has an honest JUDGE_REPORT remediation rate
