# Evaluate Skill

## Purpose
Simulate what a real hackathon judge would think. Find the fatal weaknesses before the judge does. Generate a scored `JUDGE_REPORT.md` and a concrete remediation plan.

This is the **most important skill in HackFork**. It is non-negotiable — run it before any pitch attempt.

## When to Use
- Before preparing the pitch (call this before the `pitch` skill)
- After completing Phase 1 of TODO.md
- Whenever you feel uncertain about whether the project is good enough
- When you want an objective gut-check mid-build

## Instructions

### Step 1: Activate Judge Mode
Switch to `[MODE: JUDGE]` and announce it with weight:

> "Switching to Judge Mode. I am now the skeptical, demanding, rubric-driven jury member who has seen this kind of project fail before. I will not be gentle. Let's find out if this project actually has what it takes."

### Step 2: Read All Available Context
Read the following files from the current directory in this order:
1. `PROJECT_BRIEF.md` — What the team claimed they'd build
2. `TODO.md` — What the team planned to do
3. `README.md` — What the team says they built
4. Source code / entry point files — What the team actually built
5. `git log --oneline -20` — How much real progress has been made

If any of these files are missing, note it as a finding ("Missing README is itself a judge red flag").

### Step 3: Score Against the Judging Rubric

Score each criterion on a scale of 0–25 (matching the actual weighting):

**Agent Quality (30% → max 30 points)**
Questions to evaluate:
- Is the agent's identity (SOUL.md) compelling and distinctive?
- Are the rules well-defined and specific (not vague platitudes)?
- Does the agent actually do something useful and non-trivial?
- Is the agent's behavior consistent with its stated identity?

**Skill Design (25% → max 25 points)**
Questions to evaluate:
- Are the skills focused (one skill = one capability)?
- Is each SKILL.md well-documented with clear input/output contracts?
- Do the skills compose naturally into a workflow?
- Are skills reusable beyond this specific hackathon?

**Working Demo (25% → max 25 points)**
Questions to evaluate:
- Can the demo be run in under 2 minutes with zero setup?
- Is there a clear, memorable "wow moment" in the demo?
- Does the demo show the agent making real decisions (not just string formatting)?
- Is the demo resilient to failure (fallback demo mode if API is down)?

**Creativity (20% → max 20 points)**
Questions to evaluate:
- Does this surprise the judges in a good way?
- Is the use case novel (not another "AI memo writer" or "AI trip planner")?
- Is the agent's personality distinctive and coherent?
- Is there a "story" that makes this memorable after 20 other demos?

### Step 4: Find the Top 3 Failure Modes
After scoring, identify the 3 most likely reasons this project would *lose*. Be specific. Not "the demo could be better" but "the demo has no clear climax — judges will tune out by minute 3 because there's no moment of surprise."

### Step 5: Generate JUDGE_REPORT.md
Write the following file to the current directory:

```markdown
# JUDGE_REPORT.md

> Generated by HackFork Evaluate Skill — [MODE: JUDGE]
> Timestamp: [current time]

## Overall Score: [X]/100

## Criterion Scores
| Criterion | Weight | Score | Max |
|---|---|---|---|
| Agent Quality | 30% | X | 30 |
| Skill Design | 25% | X | 25 |
| Working Demo | 25% | X | 25 |
| Creativity | 20% | X | 20 |
| **TOTAL** | 100% | **X** | **100** |

## Score Interpretation
- 90–100: 🏆 Podium-ready. Don't change a thing.
- 75–89: 🥈 Competitive. Fix the top weakness and you're there.
- 60–74: 🟡 Needs work. The core is solid but the demo or story is weak.
- 45–59: 🔴 At risk. Fundamental issue — see failure modes.
- Below 45: ⛔ Rebuild the pitch/demo from scratch.

## Top 3 Strengths
1. [Specific strength with evidence]
2. [Specific strength with evidence]
3. [Specific strength with evidence]

## Top 3 Failure Modes
1. **[Failure Mode Title]** — [Specific description of why this would cause a judge to score lower, with the criterion affected]
2. **[Failure Mode Title]** — [Same]
3. **[Failure Mode Title]** — [Same]

## Remediation Checklist
- [ ] Fix: [Specific action to address Failure Mode 1] — Est: Xh
- [ ] Fix: [Specific action to address Failure Mode 2] — Est: Xh
- [ ] Fix: [Specific action to address Failure Mode 3] — Est: Xh

## Judge Simulation: Likely Questions
1. **"[Question a skeptical judge would ask]"**
   → Suggested answer: [How to respond]
2. **"[Question]"**
   → Suggested answer: [Response]
3. **"[Question]"**
   → Suggested answer: [Response]

## Verdict
[One paragraph summary of the project's competitive position and the single most important thing to fix before presenting]
```

### Step 6: Switch Back to Builder Mode
Announce the switch:

> "Switching back to [MODE: BUILDER]. Here's what we're going to fix, and in what order..."

Then provide a specific, time-ordered remediation plan tied to the checklist in `JUDGE_REPORT.md`.

### Step 7: Suggest Next Skill
After all remediations are acknowledged: "Run the **pitch** skill to generate your demo script and presentation materials."

## Output
- File: `JUDGE_REPORT.md` (written to current directory)
- Mode: Judge → Builder transition
- Remediation plan with time estimates

## Quality Criteria
A good JUDGE_REPORT.md from this skill:
- Has a specific, evidence-based score (not a round number like "60/100" without justification)
- Has failure modes that reference specific things in the codebase or brief — not generic hackathon advice
- Has a remediation checklist where every item is completable in under 2 hours
- Has judge questions that the user hasn't already thought of
- Has a verdict that is memorable and memorable — the kind of thing a team repeats to each other
