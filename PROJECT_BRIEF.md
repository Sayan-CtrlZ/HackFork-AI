## Project Name
**HackFork** — The Adversarial Hackathon Co-Pilot

## One-Line Pitch
An AI agent with two voices — one that helps you build, one that tells you why you'll lose — so you can fix it before a judge does.

## Target User
**Alex**, a solo developer or 2-person team entering a 24-hour hackathon. Alex has built things before but always feels like the pitch falls flat or the demo has a surprise failure at the worst moment. Alex is good at building but bad at stress-testing their own ideas.

## Core Problem
Hackathons fail not because people can't code — they fail because:
1. Ideas don't get validated until the demo moment reveals the flaw
2. Code gets written before the story is clear  
3. No one challenges the team's assumptions from the outside before judges do
4. The pitch is built on what was *started*, not what was *finished*

## Solution
HackFork is a git-native AI agent with a mandatory adversarial evaluation step built into its skill pipeline. It plays Builder and Judge simultaneously — helping you build while constantly pressure-testing your assumptions against the judging rubric.

Running `ideate → scaffold → evaluate → pitch → retrospect` produces 6 real files in your repo that track your project from idea to post-mortem.

## Key Differentiator
Every other AI coding tool is a **yes machine**. HackFork is the first tool with a built-in **no machine** — an adversarial Judge persona that will tell you your project is weak before a jury member does.

The Judge persona isn't optional. It's enforced by RULES.md.

## The Demo Moment
The judge (audience) watches HackFork switch from Builder to Judge mode mid-conversation, generate a JUDGE_REPORT.md that scores the project being demoed as 87/100, identify 3 failure modes with specific fixes, then switch back to Builder and fix them.

**The punchline**: the JUDGE_REPORT.md in this repository was generated by HackFork evaluating *itself*. The meta-loop is the demo.
